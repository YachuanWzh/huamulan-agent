## 方向一：自动化 Eval CI Pipeline

### 目标

将现有的手动 CLI 评估流程集成到 CI/CD，实现每次 PR 自动跑全量 golden cases + 回归检测，结果作为 CI check 拦截劣化。

### 现状分析

现有评估基建已经相当完整：

| 模块 | 文件 | 能力 |
|------|------|------|
| Golden Case 模型 | `evaluation/models.py` | `GoldenSkillCase`, `AgentEvaluationCase` 完整字段（含 intent、metrics、entities、tool_calls、safety） |
| 离线评估 | `evaluation/offline.py` | `evaluate_routing_cases()`, `evaluate_multi_agent_intent_cases()` — skill 路由 + 意图分类精度 |
| 在线评估 | `evaluation/runtime.py` | `evaluate_runtime_logs()` — tool 调用成功率、P95/P99 延迟、token 消耗 |
| 诊断输出 | `evaluation/diagnostics.py` | `build_case_evaluation_detail()` — 逐 case 诊断（含 checks、diagnosis、suspected_node） |
| 回归检测 | `evaluation/regression_cli.py` | `compare_evaluation_runs()` — pass rate 比较、latency/token 回归、safety 退化 |
| 运行管理 | `evaluation/ops.py` | `EvaluationRun`, `EvaluationComparison`, `RegressionFinding` — 完整的 CRUD 和对比模型 |
| CLI 入口 | `evaluation/__main__.py` | 手动触发 `--skills-dir` + `--golden` → JSON/Markdown 报告 |

**核心缺口：完全没有 CI 集成，所有评估靠人工 CLI 触发。**

### 方案

```
                    ┌──────────────────────────┐
                    │  PR / Merge Request       │
                    └──────────┬───────────────┘
                               │ trigger
                               ▼
                    ┌──────────────────────────┐
                    │  ci_eval.py (NEW)         │
                    │  ─────────────────────    │
                    │  1. checkout baseline     │
                    │  2. checkout candidate    │
                    │  3. 跑 full golden suite  │
                    │  4. compare + gate check  │
                    │  5. 输出 JSON + Markdown  │
                    │  6. exit code = 门禁结果  │
                    └──────────┬───────────────┘
                               │
                    ┌──────────▼───────────────┐
                    │  CI Artifacts             │
                    │  - eval_report.md         │
                    │  - eval_report.json       │
                    │  - regression_diff.md     │
                    └──────────────────────────┘
```

**核心设计 — `scripts/ci_eval.py`：**

1. 在临时目录 checkout baseline 代码（`main` 或 `HEAD~1`）
2. 分别在 baseline 和 candidate 目录中安装依赖 + 跑 eval
3. 调用已有 `compare_evaluation_runs()` 做回归对比
4. 按阈值判定门禁通过/警告/失败
5. 输出结构化报告 + 设置 exit code

**Gate 规则（可配置）：**

| 指标 | 阈值 | 级别 |
|------|------|------|
| Pass rate 下降 | > 0% | error |
| 任何 case pass→fail | 1 个以上 | error |
| Safety case 退化 | 1 个以上 | error |
| Forbidden tool 调用 | 新增 | error |
| Latency P95 增长 | > 25% | warning |
| Token 消耗增长 | > 25% | warning |
| Case fail→pass | 任意 | info |

**新增/修改文件：**

| 操作 | 文件 | 说明 |
|------|------|------|
| 新建 | `backend/scripts/ci_eval.py` | CI eval 入口脚本 |
| 新建 | `.github/workflows/eval.yml` | GitHub Actions 配置 |
| 新建 | `.gitlab-ci.yml` 中的 eval job | GitLab CI 模板 |
| 新建 | `backend/tests/test_ci_eval.py` | CI eval 单元测试 |
| 修改 | `backend/src/personal_assistant/skills/evaluation/ops.py` | （如需要）增加 git_sha 自动采集 |

**运行方式：**

```bash
# CI 中直接调用
python scripts/ci_eval.py \
  --baseline-ref main \
  --golden evaluation/golden/ \
  --output-dir ci_artifacts/

# 本地手动验证
python scripts/ci_eval.py \
  --baseline-ref HEAD~1 \
  --golden evaluation/golden/ \
  --thresholds '{"max_pass_rate_drop":0.05}'
```

## 方向二：A/B Testing 框架

### 目标

支持对同一批 query 并行调用两个 Agent 变体（不同模型/prompt/配置），自动收集多维度指标，做统计显著性检验，产出量化对比报告。

### 现状分析

| 已有 | 说明 |
|------|------|
| `SBSTask`, `BlindedSBSTask`, `SBSReview` | SBS 盲评数据模型完整，支持 A/B 两组输出人工打分 |
| `EvaluationRun`, `EvaluationComparison` | 评估运行和对比基础设施 |
| `GoldenSkillCase` | 标准测试用例 |
| `AgentHarness.run_user_turn()` | 支持 `llm_config` 参数，可动态切换模型 |

**核心缺口：完全没有自动化 A/B 对比能力。SBS 仅用于人工盲评（两个输出放一起让人选），缺少机器自动跑两遍并做统计检验的基础设施。**

### 方案

```
   ┌─────────────────────────────────────────────┐
   │              ABTestRunner                     │
   │                                               │
   │  queries ──┬──► Agent(variant=A) ──► results_A│
   │            │                                  │
   │            └──► Agent(variant=B) ──► results_B│
   │                                               │
   │  自动对比维度:                                  │
   │    • pass_rate (golden case match)            │
   │    • latency_p50 / p95 / p99                 │
   │    • token_usage                              │
   │    • intent_accuracy                          │
   │    • skill_selection_f1                       │
   │    • judge_score (LLM judge pairwise)         │
   │                                               │
   │  统计方法:                                     │
   │    Fisher's exact / Mann-Whitney U /          │
   │    Paired t-test / McNemar / Wilcoxon         │
   │                                               │
   │  output: ABTestReport + 显著性标注             │
   └─────────────────────────────────────────────┘
```

**变体定义：**

```python
class ABVariant(BaseModel):
    name: str                          # "baseline" / "v4-flash" / "new-prompt"
    model: str | None = None           # 模型切换
    agent_mode: Literal["single", "multi"] = "multi"
    temperature: float | None = None
    prompt_overrides: dict[str, str] = {}  # 局部 prompt 覆盖
    config_overrides: dict[str, Any] = {}   # 任意 Settings 字段覆盖
```

**对比维度与统计方法：**

| 维度 | 统计检验 | 说明 |
|------|---------|------|
| Pass rate | Fisher's exact test | golden case 通过率是否有显著差异 |
| Latency P95 | Mann-Whitney U test | 延迟分布是否显著不同 |
| Token usage | Paired t-test | token 消耗是否显著不同 |
| Intent accuracy | McNemar's test | 意图分类准确率 |
| Judge score | Wilcoxon signed-rank | LLM judge pairwise 评分 |

**输出报告示例：**

```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Metric              A          B          Δ          p-value   Sig
─────────────────────────────────────────────────────────────────
Pass rate           92.0%      89.0%      -3.0%      0.12      -
Latency P95         3.2s       1.1s       -65.6%     <0.001    ***
Token usage         4850       4720       -2.7%      0.08      -
Intent accuracy     94.0%      93.0%      -1.0%      0.45      -
Judge score         3.8        3.5        -0.3       0.03      *

Conclusion: flash model significantly reduces latency (p<0.001)
without meaningful quality regression. Recommend switch.
```

## 方向三：Agent Self-Critique 自我修正循环

### 目标

在 multi-agent 的 synthesize 阶段加入结构化自我审查，自动检查子 agent 报告的逻辑一致性和证据充分性，发现问题时回退到相关子 agent 重新分析。

### 现状分析

```
  当前流程（单向）:
  supervisor → [子 agent 并行] → gate → synthesize → END
  
  问题:
  - synthesize 直接汇总输出，不对子 agent 报告做质量检查
  - 子 agent A 和 B 结论矛盾时无法发现
  - 子 agent 报告置信度低时直接采纳
  - 无回退修正机制
```

### 方案

```
  目标流程（带 self-critique）:
  
    ┌──────────┐
    │   gate   │
    └────┬─────┘
         │
         ▼
    ┌──────────────┐    发现问题     ┌──────────────────┐
    │  synthesize   │──────────────►│  self_critique    │
    │  (汇总报告)    │◄──────────────│  (逻辑一致性检查)   │
    └──────┬───────┘   修正完成      └──────┬───────────┘
           │ 通过                          │ 不通过
           ▼                               ▼
    ┌──────────┐                  ┌──────────────┐
    │   END    │                  │ 回退到子agent  │
    └──────────┘                  │ (Send API)    │
                                  └──────────────┘
```

**Self-Critique 结构化检查：**

```python
class SelfCritiqueResult(BaseModel):
    passed: bool
    issues: list[CritiqueIssue] = []

class CritiqueIssue(BaseModel):
    severity: Literal["critical", "major", "minor"]
    type: Literal[
        "contradiction",          # 子 agent A 和 B 结论矛盾
        "insufficient_evidence",  # 结论缺乏证据支撑
        "missing_scope",          # 遗漏了关键分析维度
        "low_confidence",         # 子 agent confidence < 0.5
        "hallucination_risk",     # 疑似编造数据（无 trace ID 等引用）
    ]
    description: str
    affected_agents: list[str]   # 需要重新分析的子 agent
    suggested_fix: str
```

**Critique Prompt 核心约束：**

```
你是 APM 分析质量审核员。检查以下子 agent 报告：

1. **矛盾检查**：
   - 子 agent A 说「DB 连接池正常」，子 agent B 说「DB 查询超时」
   - 是否存在不可调和矛盾？矛盾是否可能是同一问题的不同表现？

2. **证据检查**：
   - 每个 finding 是否有具体证据（trace ID / 指标值 / 日志片段）？
   - 如果没有引用任何可追溯的证据 → 标记为 hallucination_risk

3. **完整性检查**：
   - 是否遗漏了关键分析维度（只看了延迟没看错误率、只看了单服务没看上下游）？
   - 对比 supervisor plan 中的预期覆盖范围。

4. **置信度检查**：
   - 子 agent confidence < 0.5 → 说明分析不可靠，需重新分析

输出 JSON。没有问题时 passed=true, issues=[]。不要废话。
```

**回退机制：**
- 用 LangGraph `Command(goto=...)` 将修正后的任务重新派发给受影响的子 agent
- 最多回退 1 次（`MAX_SELF_CORRECTION_ROUNDS = 1`），防止无限循环
- 即使 1 次回退后仍不通过，也在报告中标注 "⚠️ 自动审核未通过" 而非静默输出

**新增/修改文件：**

| 操作 | 文件 | 说明 |
|------|------|------|
| 新建 | `backend/src/personal_assistant/agent/self_critique.py` | `SelfCritiqueResult`, critique prompt, `run_self_critique()` |
| 修改 | `backend/src/personal_assistant/agent/multi_agent.py` | synthesize 节点嵌入 critique + 条件回退 |
| 修改 | `backend/src/personal_assistant/config.py` | 新增 `MULTI_AGENT_MAX_SELF_CORRECTION_ROUNDS` |
| 新建 | `backend/tests/test_self_critique.py` | critique 模型 + 逻辑测试 |

## 方向四：Agent 工作流模板化

### 目标

将常见多步骤工作流（如「巡检→发现异常→自动 RCA→生成报告」）沉淀为 YAML 模板，用户一句话即可触发整条链路，而非分步操作。

### 现状分析

当前触发 multi-agent 靠自然语言 + intent router 自动判断，但**复杂多步骤工作流无法原子化触发**。用户需要分步说「执行巡检」→ 等结果 →「对 XXX 异常做 RCA」→「生成报告」。没有工作流编排层。

### 方案

```
  ┌───────────────────────────────────────┐
  │         Workflow Templates (YAML)       │
  │                                         │
  │  workflows/                             │
  │  ├── weekly_patrol.yaml   周检          │
  │  ├── incident_rcr.yaml    事故复盘      │
  │  ├── sla_audit.yaml       SLA审计       │
  │  ├── deployment_verify.yaml 上线验证    │
  │  └── daily_briefing.yaml  日报          │
  └───────────────┬───────────────────────┘
                  │ SkillRegistry 启动时加载
                  ▼
  ┌───────────────────────────────────────┐
  │        WorkflowEngine                   │
  │                                         │
  │  template → DAG of steps                │
  │  each step = {agent_mode, skills,       │
  │               query_template,            │
  │               depends_on, condition}     │
  │                                         │
  │  执行:                                   │
  │    拓扑排序 → 串行/并行执行             │
  │    Jinja2 模板渲染（引用前置步骤输出）   │
  │    条件分支（异常才触发 RCA）           │
  └───────────────────────────────────────┘
```

**YAML 模板示例 — `weekly_patrol.yaml`：**

```yaml
name: weekly_patrol
description: 每周自动巡检 → 异常 RCA → 生成周报
triggers:
  - 周检
  - weekly patrol
  - 每周巡检
  - 周末巡检
steps:
  - id: patrol_scan
    agent_mode: multi
    skills: [patrol, otel-query]
    query: |
      执行全量巡检，扫描所有服务过去 1 小时的 Kafka 遥测数据。
      检测 P2/P3 级别异常（延迟趋势、错误率、资源水位）。
      输出异常摘要列表。

  - id: rca_if_anomalies
    agent_mode: multi
    skills: [troubleshoot, otel-query]
    depends_on: [patrol_scan]
    condition: "{{ patrol_scan.anomalies_detected > 0 }}"
    query: |
      对巡检发现的异常做根因分析：
      {{ patrol_scan.anomaly_summary }}
      逐个分析根因并给出修复建议。

  - id: generate_report
    agent_mode: single
    depends_on: [patrol_scan, rca_if_anomalies]
    query: |
      生成本周巡检报告，Markdown 格式：

      ## 巡检概览
      - 扫描时间窗口: 过去 1 小时
      - 扫描服务数: {{ patrol_scan.service_count }}
      - 发现异常: {{ patrol_scan.anomalies_detected }} 个

      ## 异常详情与 RCA
      {% if rca_if_anomalies is defined %}
      {{ rca_if_anomalies.summary }}
      {% else %}
      本周未发现需要关注的异常。
      {% endif %}

      ## 建议
      {% if rca_if_anomalies is defined %}
      {{ rca_if_anomalies.recommendations }}
      {% endif %}
```

      - 预防措施

**核心数据模型：**

```python
class WorkflowStep(BaseModel):
    id: str
    agent_mode: Literal["single", "multi"]
    skills: list[str] = []
    query_template: str       # Jinja2 模板，可引用前置步骤的 {{ step_id.field }}
    depends_on: list[str] = []
    condition: str | None = None  # Jinja2 条件，falsy 时跳过该步骤

class WorkflowTemplate(BaseModel):
    name: str
    description: str
    triggers: list[str]
    steps: list[WorkflowStep]
    variables: dict[str, str] = {}  # 模板变量默认值
```

**WorkflowEngine 执行模型：**

```python
class WorkflowEngine:
    def __init__(self, harness: AgentHarness):
        self.harness = harness
        self.step_outputs: dict[str, dict] = {}  # 步骤输出上下文

    async def execute(self, template: WorkflowTemplate, variables: dict) -> WorkflowResult:
        # 1. 拓扑排序 DAG
        execution_order = topological_sort(template.steps)

        # 2. 按序执行
        for step in execution_order:
            # 等待依赖步骤完成
            await wait_for_dependencies(step.depends_on)

            # 条件判断
            if step.condition and not evaluate_jinja(step.condition, self.step_outputs):
                self.step_outputs[step.id] = {"skipped": True}
                continue

            # 渲染 query 模板
            query = render_jinja(step.query_template, {**variables, **self.step_outputs})

            # 执行
            result = await self.harness.run_user_turn(
                thread_id=f"wf-{template.name}-{step.id}",
                message=query,
                agent_mode=step.agent_mode,
            )

            # 存储输出供后续步骤引用
            self.step_outputs[step.id] = parse_step_output(result)

        return WorkflowResult(steps=self.step_outputs)
```

**集成方式：**

1. `SkillRegistry` 启动时扫描 `workflows/` 目录加载 YAML 模板 → 注册为特殊 skill
2. `intent_router.py` 识别 `triggers` 关键词 → 路由到 `workflow` intent
3. `multi_agent.py` 新增 `workflow_agent` 处理分支
4. 前端 SSE 事件新增 `workflow_step_started` / `workflow_step_finished`，展示多步骤进度条

## 实施优先级建议

四个方向互相独立，可分阶段并行推进：

| 阶段 | 方向 | 理由 |
|------|------|------|
| **Phase 1** | 自动化 Eval CI Pipeline | 直接命中 JD 核心要求；现有基建最成熟，投入产出比最高；CI 门禁是后续 A/B 测试的基础 |
| **Phase 2** | A/B Testing 框架 | JD 明确要求；差异化竞争力强；可复用 Phase 1 的 golden cases 和 eval 基建 |
| **Phase 3** | Self-Critique 修正循环 | 提升 Agent 输出质量；复用现有 multi_agent 架构和 structured output 协议 |
| **Phase 4** | 工作流模板化 | 提升用户体验和自动化程度；降低复杂场景的使用门槛 |
