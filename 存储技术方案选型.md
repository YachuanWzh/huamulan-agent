## 1. 架构总览

```
┌─────────────────────────────────────────────────────────┐
│                     PostgreSQL                          │
│  权威数据源 · ACID · 持久化 · 不可丢失                    │
│  checkpoint 归档 / 审计日志 / 执行日志 / 长期记忆 / 工具结果 │
└──────────┬──────────────────────────────┬───────────────┘
           │                              │
      ┌────▼────────┐              ┌──────▼────────┐
      │    Redis     │              │    Qdrant     │
      │  可选缓存     │              │  可选向量库    │
      │  可丢失       │              │  可丢失        │
      │  加速热路径    │              │  语义检索      │
      └─────────────┘              └───────────────┘
```

**核心原则**：PostgreSQL 是唯一的 source of truth，Redis 和 Qdrant 都是**可选、可丢失**的加速层。任一不可用时系统自动降级，核心功能零影响。

## 2. PostgreSQL — 权威数据源

### 2.1 定位

PostgreSQL 是项目**唯一强制依赖**的存储后端。所有需要持久化、不可丢失的数据最终落盘到 PostgreSQL。

### 2.2 数据表与职责

| 表名 | 用途 | 写入频率 |
|------|------|----------|
| `checkpoints` | LangGraph Agent 状态快照（ReAct 循环回放的核心） | 每轮多次 |
| `checkpoint_writes` | 每个 checkpoint 内的待处理通道写入 | 每轮多次 |
| `checkpoint_blobs` | Checkpoint 负载的序列化二进制数据 | 每轮多次 |
| `audit_events` | 安全审计事件（Prompt Guard / Tool Guard 拦截、审批决策） | 按事件 |
| `agent_execution_logs` | 结构化执行链路（turn / LLM / tool / retry / approval / security） | 每轮多次 |
| `tool_errors` | 工具执行失败记录，含重试次数与错误详情 | 按失败 |
| `tool_results` | 工具调用返回值，供上下文压缩后反查 | 按调用 |
| `long_term_memories` | 用户审批确认后沉淀的长期记忆（slug/key 索引，upsert） | 低频 |

所有表由 `PostgresMemory` 在服务启动时通过 `CREATE TABLE IF NOT EXISTS` 自动创建，无需手动迁移。

### 2.3 选型依据

| 维度 | 依据 |
|------|------|
| **框架集成** | LangGraph 原生提供 `langgraph-checkpoint-postgres`（`AsyncPostgresSaver`），与 StateGraph 的 checkpoint 机制深度集成，无需自建 |
| **数据模型匹配** | 审计日志、执行日志天然适合关系模型——按 `thread_id` 关联、按时间排序、按事件类型筛选，SQL 表达能力直接高效 |
| **JSONB 灵活扩展** | `agent_execution_logs` 的 `input`/`output`/`token_usage`/`metadata` 字段使用 JSONB，兼顾结构化查询与半结构化灵活存储。执行摘要 API 用单条 SQL 聚合查询即可完成事件计数、token 求和、错误率统计 |
| **ACID 强一致性** | 审批状态、安全事件、checkpoint 写入不可接受丢失或脏读。PostgreSQL 的事务保证是审计合规的基础 |
| **生态成熟** | 运维工具丰富、监控方案完善、团队知识储备充分 |

## 3. Redis — 可选加速层

### 3.1 定位

Redis 是**纯加速层**，可丢失。`REDIS_URL` 未配置时自动使用 `NoopCache`，业务功能完全不受影响。

Redis 在本项目中承担**两个独立角色**：读缓存（Role A）和 Redis-first Checkpoint 写入缓冲（Role B）。

### 3.2 Role A：读缓存（API Acceleration）

**控制开关**：`CACHE_ENABLED` + `REDIS_URL`

`CachedPostgresMemory`（组合模式包装 `PostgresMemory`，非继承）对以下读路径做 TTL 缓存：

| 缓存内容 | Key 模式 | TTL | 原因 |
|----------|----------|-----|------|
| 线程列表 | `pa:v1:threads:list:{limit}` | 10s | 侧边栏高频刷新，允许短暂不一致 |
| 执行摘要 | `pa:v1:execution_summary:{thread_id}` | 10s | 聚合计算较重（多表 JOIN + COUNT/SUM） |
| 执行日志 | `pa:v1:execution_logs:{thread_id}:{limit}` | 5s | 写入频繁，短 TTL 保证新鲜度 |
| 审计事件 | `pa:v1:audit_events:{scope}:{limit}` | 10s | 低频但可缓存 |
| 工具错误 | `pa:v1:tool_errors:{scope}:{limit}` | 10s | 低频但可缓存 |
| 长期记忆拼接 | `pa:v1:long_term_memory:{hash}` | 60s | Markdown 文件拼接耗时，按文件 mtime/size 版本化 |

**缓存失效策略**：写入 PostgreSQL 后主动删除对应 Redis key：
- `record_execution_log` → 删除该线程的 `execution_summary`、`execution_logs:*`、`threads:list:*`
- `record_audit_event` → 删除对应 `audit_events:*`
- `record_tool_error` → 删除对应 `tool_errors:*`
- `delete_thread` / `clear_threads` → 删除线程级所有缓存 + 列表缓存

### 3.3 Role B：Redis-first Checkpoint 写入缓冲

**控制开关**：`REDIS_URL` + `CHECKPOINT_TTL_SECONDS`

这是提交 `a2341ab` 实现的核心优化，引入 `RedisFirstCheckpointSaver` 包装 `AsyncPostgresSaver`：

#### 写入路径

```
LangGraph 图节点产生 checkpoint
  │
  ├─ 确定性节点（route_skills / compact_context）→ 跳过，不写 Redis 也不写 PostgreSQL
  │
  └─ 非确定性节点（agent / tools / approval / memory_reflection）
       │
       ├─ 1. CompressedJsonPlusSerializer 序列化（MessagePack + zlib 压缩）
       ├─ 2. 同步写入 Redis（SET key value EX CHECKPOINT_TTL_SECONDS）
       ├─ 3. asyncio.create_task 异步写入 PostgreSQL
       │
       └─ Redis 写入失败 → 降级为同步写 PostgreSQL（不丢状态）
```

#### 读取路径

- `aget_tuple`：先查 Redis → miss 后回退 PostgreSQL
- `alist`：Redis Sorted Set `ZREVRANGE` 最新优先 → 无数据时回退 PostgreSQL

#### 序列化

使用 `CompressedJsonPlusSerializer`（包装 LangGraph 的 `JsonPlusSerializer`）：
- 底层：MessagePack 编码
- 压缩：zlib（payload > 1024 字节时触发）
- 前缀：版本化标记字节，反序列化自动识别压缩/未压缩格式
- 单条 Redis key 存储完整信封：config + payload + metadata + versions + parent_config + timestamp + node_name

#### 选择性 Checkpoint

通过 `CHECKPOINT_SKIP_NODES` 跳过确定性节点：

| 跳过（可重算） | 保留（有外部交互或关键状态变化） |
|---------------|--------------------------------|
| `route_skills` | `agent` |
| `compact_context` | `tools` |
| | `approval` |
| | `memory_reflection` |

日志中的 `source=input/loop` 是 LangGraph checkpoint 来源元数据，**不参与** skip 判断。

#### LRU 淘汰

启动时 best-effort 执行 `CONFIG SET maxmemory-policy allkeys-lru`。托管 Redis 服务可能禁止 `CONFIG SET`，失败仅警告不阻塞启动。

### 3.4 选型依据

| 维度 | 依据 |
|------|------|
| **亚毫秒级读取** | 内存存储，checkpoint 热路径读取远快于 PostgreSQL 磁盘 I/O |
| **内置 TTL** | `SET ... EX` 自动过期，无需手动清理旧 checkpoint |
| **Sorted Set** | `ZREVRANGE` 实现时间排序的 checkpoint 列表，天然匹配回放需求 |
| **读写分离** | 近期 checkpoint 走 Redis（微秒级），历史 checkpoint 走 PostgreSQL（毫秒级），自然冷热分层 |
| **LRU 自动淘汰** | `allkeys-lru` 策略自动淘汰冷数据，结合 TTL 无需容量规划 |
| **非强依赖** | Redis 不可用时，缓存退回 `NoopCache`，checkpoint 退回同步 PostgreSQL，零功能影响 |
| **现有 commit 演进** | `33cbd78` 先引入读缓存 → `a2341ab` 扩展为 Redis-first checkpoint，增量演进而非一次大重构 |

## 4. Qdrant — 可选向量索引

### 4.1 定位

Qdrant 仅服务于 **Skill 语义路由的第二层（向量相似度召回）**。仅在三个条件同时满足时启用：

- `SKILL_ROUTING_SEMANTIC_ENABLED=true`
- `SKILL_ROUTING_VECTOR_STORE=qdrant`
- `SKILL_ROUTING_QDRANT_URL` 已配置

默认使用 `memory` 模式（进程内 Python dict），不依赖外部向量数据库。

### 4.2 职责：三层路由漏斗

```
用户输入
  │
  ├─ ① 显式 regex/触发词匹配（始终启用）
  │    命中 → 短路返回，不调用 embedding 或 LLM
  │
  ├─ ② bge-m3 embedding + 向量相似度召回（需启用语义路由）
  │    ├─ memory 模式：进程内 dict，重启丢失
  │    └─ qdrant 模式：持久化向量，多进程共享
  │    分数 ≥ 0.72 → 直接选中
  │
  └─ ③ LLM judge（分数 < 0.72 但有候选时）
       将 userInput + relatedFind 交给 LLM 结构化判断
```

### 4.3 增量同步机制

`QdrantSkillVectorIndex` 在服务启动时预热：

1. `scroll` 读取现有 Qdrant collection 中的所有 point（含 payload）
2. 对比每个 Skill 的 `source_hash`（`SKILL.md` 内容哈希）
3. 仅对新增或变化的 Skill 调用 Ollama `bge-m3` 重新生成 embedding
4. `upsert` 更新到 Qdrant（点 ID 使用 UUIDv5，namespace URL + skill 路径，确定性可重复）
5. 预热失败不阻塞启动，用户请求时懒同步兜底

### 4.4 选型依据

| 维度 | 依据 |
|------|------|
| **持久化向量** | `memory` 模式重启丢失所有 embedding，需重新生成（Ollama 生成有延迟）。Qdrant 持久化后仅增量更新 |
| **HNSW 索引性能** | Qdrant 的 HNSW 索引专为高维向量 ANN 搜索设计，远优于 PostgreSQL `pgvector` 或自建方案 |
| **Payload 过滤** | 按 `skill_name`、`source_hash` 过滤和更新，支持增量同步逻辑，避免每次全量重新 embedding |
| **多进程共享** | 局域网部署（`http://<qdrant-host>:6333`），多服务实例共享同一向量索引 |
| **REST API** | HTTP API 集成简单，无需 gRPC 额外依赖 |
| **嵌入与存储解耦** | Ollama `bge-m3` 生成向量，Qdrant 存储和检索，两者可独立替换 |
| **可选架构** | 语义路由本身可选（默认关闭），Qdrant 又是语义路由的可选后端——双层可选，不增加本地开发复杂度 |

## 5. 降级策略与容错

三种存储遵循统一的容错原则：**外层可选组件不可用时，自动降级为内层必选组件，业务不中断。**

| 故障场景 | 降级行为 | 影响 |
|----------|----------|------|
| PostgreSQL 不可用 | **整个应用不可用** | **致命** |
| Redis 未配置 | 缓存 → `NoopCache`（始终 miss）<br>Checkpoint → 同步直写 PostgreSQL | 性能下降，功能正常 |
| Redis 运行时宕机 | 读 miss → 回退 PostgreSQL/文件系统<br>写失败 → 同步写 PostgreSQL<br>日志记录 `cache_error` 事件 | 性能下降，功能正常 |
| Redis LRU 配置失败 | 警告日志，不阻塞启动 | 无 |
| Qdrant 未配置 | 向量索引 → `InMemorySkillVectorIndex` | 重启丢失 embedding |
| Qdrant 运行时不可用 | 语义路由第二层不可用 → 退回纯 regex 路由 | 部分 Skill 无法语义匹配 |
| Ollama 不可用 | embedding 生成失败 → 语义路由第二层不可用 | 同 Qdrant 不可用 |

## 6. 环境变量速查

### PostgreSQL（必选）

```ini
DATABASE_URL=postgresql://user:password@host:5432/dbname?sslmode=disable
```

### Redis（可选 — 缓存 + Checkpoint 加速）

```ini
# 读缓存
CACHE_ENABLED=true
REDIS_URL=redis://redis.example.local:6379/0
CACHE_DEFAULT_TTL_SECONDS=10
CACHE_LOG_TTL_SECONDS=5
CACHE_MEMORY_TTL_SECONDS=60

# Redis-first Checkpoint
CHECKPOINT_TTL_SECONDS=604800
CHECKPOINT_PG_CLEANUP_ENABLED=true
CHECKPOINT_PG_CLEANUP_INTERVAL_SECONDS=3600
CHECKPOINT_REDIS_LRU_ENABLED=true
CHECKPOINT_REDIS_MAXMEMORY_POLICY=allkeys-lru
CHECKPOINT_SKIP_NODES=route_skills,compact_context
```

### Qdrant（可选 — Skill 语义路由）

```ini
SKILL_ROUTING_SEMANTIC_ENABLED=true
SKILL_ROUTING_VECTOR_STORE=qdrant
SKILL_ROUTING_QDRANT_URL=http://<qdrant-host>:6333
SKILL_ROUTING_QDRANT_API_KEY=your-api-key
SKILL_ROUTING_QDRANT_COLLECTION=skill_routes
SKILL_ROUTING_EMBEDDING_MODEL=bge-m3
SKILL_ROUTING_OLLAMA_BASE_URL=http://localhost:11434
SKILL_ROUTING_SIMILARITY_THRESHOLD=0.72
SKILL_ROUTING_TOP_K=3
```

## 7. 总结

```
本地开发          生产环境（最小）       生产环境（完整）
────────          ──────────────        ──────────────
PostgreSQL  ✅    PostgreSQL     ✅     PostgreSQL     ✅
Redis       ❌    Redis          ✅     Redis          ✅
Qdrant      ❌    Qdrant         ❌     Qdrant         ✅
                  （读缓存 + checkpoint 加速）  （+ Skill 语义路由）
```

| 存储 | 强制 | 角色 | 存储内容 | 故障后果 |
|------|------|------|----------|----------|
| **PostgreSQL** | 是 | 权威数据源 | Checkpoint 归档、审计、日志、记忆、工具结果 | **应用不可用** |
| **Redis** | 否 | 加速层 | (A) 查询缓存 (B) Checkpoint 热写入缓冲 | 性能下降，功能正常 |
| **Qdrant** | 否 | 向量索引 | Skill 描述 embedding（1024 维 bge-m3 向量） | 语义路由退化为纯 regex |

**选型核心逻辑**：PostgreSQL 满足 ACID 持久化需求，是 LangGraph 官方 checkpoint 后端；Redis 以亚毫秒级内存读写优化热路径，TTL + LRU 自动冷热分层；Qdrant 以 HNSW 索引支持高维向量语义检索。三者按"渐进式增强"原则组合——本地开发只需 PostgreSQL，生产按需叠加 Redis 和 Qdrant，任一层不可用时自动降级，核心对话功能零影响。
