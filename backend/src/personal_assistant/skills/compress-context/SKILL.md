# Compress Context — 上下文压缩

当用户手动调用本技能时，对**当前对话的完整历史**执行压缩，输出一份紧凑的摘要。
压缩后的内容可直接作为新会话的初始上下文，减少 token 消耗的同时保留关键决策链。

## 触发方式

本技能**仅支持手动触发**，无自动触发器。用户通过 `/compress-context` 或直接要求
"压缩上下文"、"总结对话" 等方式调用。

## 压缩策略

按以下四步逐层压缩，每一步的输出作为下一步的输入：

### 第 1 层：保留用户首次输入

从对话历史中提取用户的**第一条消息**（不含系统指令与 `<system-reminder>` 块），
原样保留。如果首条消息过长（超过 500 字），可截断并标注 `[…truncated]`。

### 第 2 层：保留助手首尾输出

- **首次输出**：助手的第一条实质性回复（跳过 "我将…" 之类的确认语，取实际内容）。
- **末次输出**：助手的最后一条回复，原样保留。

### 第 3 层：中间过程 LLM 摘要

对第 1 层首条用户消息之后、第 2 层末次助手输出之前的**所有中间对话**，调用 LLM
生成结构化摘要。摘要必须覆盖：

| 维度 | 说明 |
|------|------|
| **决策** | 做了哪些关键决策？选择了什么方案？为什么？ |
| **文件变更** | 创建/修改/删除了哪些文件？核心改动是什么？ |
| **问题与修复** | 遇到了什么错误？如何解决？ |
| **未完成事项** | 还有哪些任务未完成？哪些决策待定？ |
| **关键结论** | 用户确认了什么？达成了什么共识？ |

摘要使用简洁的列表/要点格式，避免冗余叙述。

### 第 4 层：组装输出

将以上三层组装为以下结构：

```
## 对话压缩摘要 (compress-context)

### 用户首次输入
[原始首条用户消息]

### 中间过程摘要
[第 3 层的 LLM 摘要]

### 助手首次输出
[助手第一条实质性回复]

### 助手末次输出
[助手最后一条回复]

---
压缩时间: {当前 ISO-8601 时间}
原始轮次: N 轮
压缩后估算 tokens: ~XXX
```

## 使用示例

```
用户: /compress-context
助手: [执行上述四步策略，输出压缩摘要]
```

## 注意事项

- 压缩摘要应**自包含**：新会话仅凭摘要即可理解当前进度，无需回溯原始对话。
- 如果对话历史过短（少于 4 轮），直接输出 "对话较短，无需压缩" 并附上简要总结。
- 工具调用结果（如文件内容、命令输出）不逐字保留，而是在摘要中提炼关键信息。
- 如果中间存在用户明确标记为 "重要" 或 "记住" 的内容，在摘要中加 **⚠️ 重要** 标记。
